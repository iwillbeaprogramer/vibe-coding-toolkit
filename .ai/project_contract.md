# Project Contract

## Hard Rules
- 모델은 Git commit, amend, reset, checkout, rebase, push를 직접 실행하지 않는다.
- 기존 테스트를 삭제하거나 비활성화하지 않는다.
- 요청 범위를 벗어난 리팩터링, 의존성 추가, 파일 이동은 하지 않는다.

## Project Layout
- 프로덕션 코드는 루트 `src/` 하위에 둔다.
- 테스트 코드는 루트 `tests/` 하위에 둔다.
- 새 테스트 파일을 `src/` 하위에 만들지 않는다.
- WPF 프론트엔드와 Python FastAPI 백엔드를 함께 사용하는 기능은 프론트엔드를 `src/<feature>-dashboard/`에, 백엔드를 `src/<feature>-api/`에 분리해 둔다.

## Code Style
- 새 코드는 같은 디렉터리의 기존 패턴, 네이밍, 파일 구조를 우선 따른다.
- 프로젝트에 이미 명확한 네이밍 관례가 없으면 변수와 함수는 camelCase를 기본으로 한다.
- 함수 이름은 가능하면 동사 또는 동사구로 시작한다. 예: `loadConfig`, `validateInput`, `renderItem`.
- Boolean 값과 Boolean 반환 함수는 `is`, `has`, `can`, `should` 같은 의미 있는 접두사를 사용한다.
- 이벤트 핸들러는 `handle` 또는 기존 프로젝트의 이벤트 네이밍 패턴을 따른다.
- 값을 변환하는 함수는 `to`, `from`, `parse`, `format`, `normalize`처럼 변환 의도가 드러나는 이름을 사용한다.
- 데이터를 가져오는 함수는 `get`, `load`, `fetch`, `read` 중 실제 동작에 맞는 동사를 사용한다.
- 부수효과가 있는 함수는 `save`, `write`, `update`, `delete`, `send`, `create`처럼 변경 의도가 드러나는 동사를 사용한다.
- 구현이 30줄을 넘어가면 함수나 작은 단위로 분리한다.
- 디렉터리 및 솔루션 이름은 단수형을 기본으로 하고, Python 패키지 이름은 하이픈 대신 언더스코어를 사용한다.

## Reliability
- 외부 입력, 파일, 네트워크, 프로세스 실행 결과는 실패 가능성을 명시적으로 처리한다.
- 백엔드 주소, 포트, 타임아웃 등 환경에 따라 달라질 수 있는 값은 코드에 하드코딩하지 않고 상수 또는 설정으로 분리해 외부에서 변경 가능하게 둔다.
- 외부 데이터 공급원(네트워크 API, 외부 라이브러리)을 사용하는 코드는 호출부를 의존성 주입 가능한 Provider 또는 인터페이스로 감싸 테스트에서 가짜 객체로 교체할 수 있게 만든다.

## Error Handling
- 백엔드 API의 오류 응답은 `{"error": {"code", "message"}}` 형태의 봉투(envelope) 구조로 통일하고, 입력 형식 위반은 400, 미존재 리소스는 404, 외부 공급원 장애는 502/503으로 상태 코드를 의미별로 구분한다.
- 외부 라이브러리/API에 전달되는 사용자 입력 파라미터는 라이브러리에 그대로 위임하지 않고, 호출 전에 허용 값 화이트리스트로 검증해 잘못된 값은 명확한 오류로 거절한다.

## UX
- UI 코드에서 비동기 외부 호출을 수행할 때는 로딩 표시와 중복 실행 방지 플래그를 두고, 네트워크 실패/타임아웃/4xx/5xx 등 오류 종류를 사용자 친화적인 메시지로 구분 표시한다.

## Testing
- 백엔드 테스트는 외부 네트워크 호출을 직접 수행하지 않는다. 외부 의존성은 DI 또는 monkeypatch로 가짜 객체로 대체한 상태에서 검증한다.
